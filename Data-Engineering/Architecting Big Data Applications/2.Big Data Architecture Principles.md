# Course Notes: Big Data Architecture Principles

## 1. Making the Choice: Real-Time vs. Batch

Deciding whether a big data pipeline should be batch or real-time is a critical foundational decision. It heavily impacts latency, infrastructure scale, total cost, manageability, and system resiliency.

While engineers are often drawn to real-time pipelines for the technical challenge, they introduce unique operational complexities:

- **Uneven Loads & Idling Costs:** Real-time pipelines are event-driven. Because source data arrives at an uneven pace, infrastructure must be provisioned for _peak load_ rather than average load to prevent latency spikes. This results in significant idle resource costs.
- **Back Pressure:** If a real-time system cannot process data at the speed of arrival, back pressure builds up, which can slow down or crash upstream services.
- **Debugging Difficulties:** Monitoring is highly complex due to transient data and states that cannot easily be isolated or reproduced for troubleshooting.
- **High Redundancy Costs:** Recovering from failures while keeping a continuous system online requires massive resource redundancy.

### The Latency Decision Framework

| Latency Requirement          | Recommended Approach  | Strategy                                                                                                                |
| ---------------------------- | --------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **Greater than 30 minutes**  | **Batch**             | Standard choice. Highly cost-effective and resilient.                                                                   |
| **Between 5 and 30 minutes** | **Batch (Preferred)** | Lean toward batch. Negotiate with users to see if they can tolerate a longer window for better business value vs. cost. |
| **Less than 5 minutes**      | **Real-Time**         | Deploy only if the business use case absolutely demands immediate data processing.                                      |

> **Alternative - Hybrid Pipelines:** If a project has both instant-action and historical requirements, build separate pipelines. Let the real-time pipeline handle only the critical, low-latency requirements, and send the rest to a batch pipeline.

## 2. Horizontal Scaling & Distributed Processing

### Horizontal Scaling

Horizontal scaling increases application capacity by adding compute nodes and distributing input data across them. Ideally, adding hardware should yield a linear capacity increase (e.g., adding 2 nodes to a 3-node cluster should boost capacity by roughly 66%).

However, system design flaws can cause performance gains to plateau due to specific architectural bottlenecks:

- **Linear Scale Operations:** Operations that process individual records independently (e.g., `map`, `filter`, `transform`, `enrich`) scale perfectly across nodes.
- **System Choke Points:**
- _Aggregations:_ Global aggregations force data onto a single node, risking memory limits and disk swapping.
- _Joins and Merges:_ These require bringing two large datasets together on a single node.
- _Database Locks:_ Forcing sequential operations at high volumes halts concurrency.

### Scaling & Distributed Processing Best Practices

- **Filter Early, Aggregate Late:** Drop unwanted records and attributes as early as possible in the pipeline to minimize downstream data volume.
- **Two-Stage Aggregations:** For global calculations, run intermediate, distributed key-based aggregations first, then summarize those smaller outputs globally.
- **Optimize Joins:** If one dataset is small or static, replace heavy joins with simple lookups or leverage data caching.
- **Decouple via Distributed Processing:** Follow microservices principles by splitting a massive job into distinct, decoupled sub-tasks based on cohesion.

## 3. Technology Selection & Integration

### The Technology Selection Process

Big data systems rely heavily on out-of-the-box components (e.g., Apache Spark for processing, Kafka for messaging, MongoDB for NoSQL, and Redis for caching). Because no single technology fits every scenario, architects must use an objective, data-driven selection process:

1. **Establish Use Case Criteria:** Under 10 Criteria.
   Create a highly specific list of requirements (functional patterns, scale, latency, concurrency, cost, maturity, and team skillset). Keep the list under 10 items.

2. **Assign Weights or Flags:** Quantify Importance.
   Apply numeric weights to each criterion based on its importance to the business, or use a Red/Orange/Green flagging system.

3. **Shortlist Technologies:** Filter Candidates.
   Identify a small group of out-of-the-box technologies that closely match your core functional and non-functional requirements.

4. **Score and Weigh Options:** Calculate Scores.
   Evaluate each shortlisted technology against your criteria. Assign individual scores, calculate the weighted totals, and choose the technology with the highest overall score.

---

> **Skilled Personnel Overhead:** Minimize the number of unique technologies across your organization. For example, maintaining six different database types creates massive resource and knowledge overhead. Only adopt a new technology if existing internal tools cannot meet the requirements.

### Achieving Optimal Technology Integration

Even if individual technologies (like Kafka, Spark, and HDFS) scale massively on their own, the data passing _between_ them can become a critical performance bottleneck.

- **Prioritize Native Connectors:** Look for built-in, out-of-the-box connectors during the evaluation phase.
- **Leverage Distributed Pipelines:** Good connectors exploit the parallel nature of both systems. For instance, a proper integration allows multiple Apache Spark task nodes to write concurrently to independent partitions on HDFS without causing database locks.
- **Understand System Internals:** Always research the underlying limitations of your selected tech stack to build integrations that maximize their native features.
