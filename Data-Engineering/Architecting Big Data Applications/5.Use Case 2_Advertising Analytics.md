# Use Case 2: Advertising Analytics

## 1. Problem Definition & Goals

### The Problem

ABC Company markets products using multiple promotional channels:

- **Email Marketing:** Direct email campaigns executed via an on-premise campaign management platform (**1.0 million emails/day**).
- **E-Commerce Display Ads:** Targeted pop-up advertisements on a cloud-hosted e-commerce site based on browsing history (**0.75 million ads/day**).

The company needs to evaluate campaign effectiveness by calculating the **Click-Through Rate (CTR)**—the ratio of ad/email impressions shown versus user views/clicks—disaggregated by **Product**, **Customer Age Group**, and **Gender**.

$$\text{Click-Through Rate (CTR)} = \frac{\text{Total Views / Responses}}{\text{Total Impressions / Offers Delivered}}$$

### Architecture Goals

- **Extensible Pipeline:** Easily integrate new advertising channels (e.g., social media or TV) in the future without refactoring existing components.
- **Daily Aggregation:** Consolidate raw data from all channels into daily summary metrics.
- **BI & SQL Integration:** Expose aggregated analytical data via standard SQL interfaces so out-of-the-box Business Intelligence (BI) tools can query insights.

---

## 2. Requirements & Baseline Workflow

### Component Analysis

- **Inputs:**
- _Cloud E-Commerce Platform:_ Exposes an API returning **JSON** records of displayed ads, demographic info, and user responses. Restricted by a **15-minute rate limit window**.
- _Email Campaign Management System:_ Exposes an intraday API providing **CSV** extracts containing email delivery, demographic, and response logs.

- **Outputs:** An analytics database storing **aggregated daily CTR metrics** for a 3-year historical retention window.
- **Target Output Schema:** `Date` | `Channel` | `Product` | `Age Group` | `Gender` | `Impressions Offered` | `Customer Responses`
- **Non-Functional Requirements:** Daily summaries must be available within 24 hours of day-end. Downtimes under 1 hour are acceptable. Authentication is required, but no private user data (PII) is stored in the summary layer.

### Baseline Functional Workflow

Because outputs are broken down by channel, the architecture deploys **decoupled, independent pipelines** for each channel to avoid shared single points of failure and allow independent scaling.

```
[Cloud E-Commerce API] ──► [Ads Extractor]   ──► [File-based Cache] ──► [Daily Summarizer] ──┐
                                                                                            ▼
                                                                                   [Analytics DB (MySQL)]
                                                                                            ▲
[Email Platform API]   ──► [Email Extractor] ──► [File-based Cache] ──► [Daily Summarizer] ──┘

```

---

## 3. Designing for Scale

### Composite Key Strategy

To distribute compute load across nodes, all extraction, caching, and summarization tasks use a **Composite Partitioning Key**:

$$\text{Composite Key} = \text{Product} + \text{Age Group} + \text{Gender}$$

- **Extraction & Caching:** Extractors divide API fetches by this composite key and write raw JSON/CSV records into directory trees formatted by key attributes.
- **Parallel Summarization:** Summarizer nodes process individual directory partitions concurrently without thread locks.
- **API Rate Limit Gate:** Because source APIs enforce strict rate-limiting (e.g., e-commerce API rate limit of once per 15 minutes), extractor worker threads must be throttled to operate safely within source thresholds.

---

## 4. Final Architecture Review

The finalized analytical pipeline utilizes custom extractors, distributed file caching, parallel compute, and a relational analytics store:

### Key Architecture Features

- **Custom API Extractors:** Lightweight custom tools adapt to unique JSON/CSV payloads and enforce API rate-limit throttling rules.
- **HDFS Temporary File Cache:** Serves as a staging ground. Data is written into HDFS folder structures keyed by `Channel/Date/Product/AgeGroup/Gender`. Once daily summarization completes, raw cache files are purged.
- **Apache Spark Summarizer:** Reads staging folders concurrently, computes daily CTR metrics across demographic combinations, and writes results to MySQL using native JDBC drivers.
- **MySQL Analytics Target:** Because raw data (~1.75M daily events) is aggregated into significantly smaller daily summary rows, a relational database like MySQL easily satisfies the 3-year retention query requirement and provides seamless connectivity for standard BI tools.

When choosing between **Amazon S3** and **HDFS (Hadoop Distributed File System)** for temporary staging or caching in batch pipelines, the core trade-off comes down to **cloud-native convenience vs. raw local performance**.

Here is a breakdown of how they compare for staging raw batch data:

---

## Direct Comparison

| Feature / Dimension      | Amazon S3 (Object Storage)                                                          | HDFS (Distributed File System)                                                       |
| ------------------------ | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| **Compute & Storage**    | **Decoupled:** Scale storage infinitely without paying for compute nodes.           | **Coupled:** Adding storage requires adding active cluster worker nodes.             |
| **Maintenance**          | Zero infrastructure management (Serverless/Managed).                                | High operational overhead (NameNode management, disk balancing, patching).           |
| **Data Locality**        | **No locality:** Data must travel over the network to compute engine (e.g., Spark). | **High locality:** Compute tasks run on the exact physical server holding the block. |
| **Directory Operations** | Metadata operations (like renaming/moving folders) are slower.                      | Atomic, near-instant directory renames and file moves.                               |
| **Cost (Idle)**          | Extremely cheap (pay only for gigabytes stored).                                    | High (must pay for running compute nodes even when data is idle).                    |

---

## 1. Amazon S3 as a Temporary Cache

### Pros

- **Decoupled Scaling & Cost Efficiency:** Because batch processing runs periodically (e.g., once a day), S3 allows you to turn off your compute cluster (like AWS EMR or Databricks) when jobs finish. You only pay pennies per gigabyte for the temporary files while they sit in S3.
- **Built-In Lifecycle Policies:** S3 allows you to set automated TTL (Time-To-Live) expiration rules on staging buckets (e.g., _"Delete all objects after 7 days"_). This completely automates raw cache pruning without writing extra maintenance scripts.
- **Infinite Elasticity:** You don't need to forecast storage capacity for sudden spikes in raw log volume—S3 scales seamlessly without disk-space errors.

### Cons

- **Network Latency & Throughput:** Reading and writing large volumes over the network introduces higher I/O latency compared to reading directly from local NVMe/SATA drives on dedicated cluster nodes.
- **API Rate Limits:** While S3 handles massive throughput, extreme parallel requests to the exact same prefix can hit partition rate limits (5,500 GETs / 3,500 PUTs per second per prefix) unless your directory keys are properly partitioned.

---

## 2. HDFS as a Temporary Cache

### Pros

- **Data Locality (Speed):** Spark tasks can be scheduled on the specific cluster node that physically stores the raw HDFS block. This eliminates network transfer bottlenecks and yields faster read/write throughput during heavy transformations.
- **Fast Directory & File Renames:** HDFS supports instant POSIX-like directory moves. In batch pipelines, moving temporary output files to a final directory is an atomic, sub-second operation (whereas S3 must copy and delete objects under the hood).
- **No API Throttling:** Throughput is bounded only by local disk I/O and internal network fabric, avoiding third-party API rate limits.

### Cons

- **High Compute Cost During Idle Time:** Because HDFS requires running DataNodes, you must pay for live compute instances 24/7—even during the multi-hour gaps between batch runs.
- **Heavy Operational Overhead:** HDFS requires active monitoring to manage disk utilization, rebalance blocks across the cluster, and handle NameNode failovers.

---

## Summary Recommendation

> **Use Amazon S3 if:** You are building in the cloud, want to spin down compute clusters between batch runs to minimize costs, and prefer automated bucket lifecycle rules to manage cache cleanup.

> **Use HDFS if:** You run an on-premise datacenter, or have a continuously running 24/7 cluster where extreme read/write throughput and sub-second file renames outweigh cloud storage savings.

Handling late-arriving data (e.g., ad impressions or email clicks delivered days after an event due to offline devices, network delays, or source API lag) is a classic challenge in batch processing.

In our **Advertising Analytics** pipeline, if we simply re-run a daily batch job using an accumulator or blindly append data, we risk either **double-counting metrics** or **overwriting accurate figures**.

Here are the three primary architectural strategies to handle late-arriving data in this pipeline, ranging from simplest to most robust:

---

## Strategy 1: Lookback Reprocessing Window (Recommended)

Instead of only running the daily summarizer for $T-1$ (yesterday), expand the daily batch job's scope to process a **sliding window** (e.g., the last $N$ days, such as 7 days).

```
[Daily Run on Day 10] ──► Summarizes & Overwrites Partitions for: [Day 9] [Day 8] [Day 7] [Day 6] [Day 5] [Day 4] [Day 3]

```

### How It Works:

1. **Raw Cache Partitioning:** Organize raw HDFS/S3 data by event occurrence date (`/staging/event_date=YYYY-MM-DD/`). Late records are dropped into their _actual event date_ folder, not the ingestion date folder.
2. **Batch Window Execution:** Every night, Apache Spark reads data for the target date **plus the prior 6 days**.
3. **Idempotent Overwrite:** Spark re-computes the aggregate CTR totals for those 7 partition dates and uses an `UPSERT` or `INSERT OVERWRITE` in the MySQL database.

### Pros & Cons

- **Pros:** Simple to implement; guarantees complete consistency for any data arriving within the window without complex delta logic.
- **Cons:** Re-computes data that hasn't changed, consuming extra compute resources.

---

## Strategy 2: Delta Ingestion & Aggregate Update (Watermarking)

If re-processing multiple days of historical data is too compute-intensive, use a **watermark / state-tracking approach** to compute and apply only the incremental differences (_deltas_).

### How It Works:

1. **Track Processed State:** Maintain a tracking table recording the last processed timestamp for every source system.
2. **Isolate Late Arrivals:** The daily extractor fetches any record where `created_at > last_processed_timestamp` OR where `event_timestamp < batch_start_date`.
3. **Additive SQL Updates:** Instead of overwriting daily totals, execute SQL `UPSERT` statements that increment existing metrics:

```sql
INSERT INTO daily_ctr_summaries
    (date, channel, product, age_group, gender, offers_count, response_count)
VALUES
    ('2026-07-15', 'email', 'WidgetA', '25-34', 'F', 100, 5)
ON DUPLICATE KEY UPDATE
    offers_count = offers_count + VALUES(offers_count),
    response_count = response_count + VALUES(response_count);

```

### Pros & Cons

- **Pros:** Highly efficient; only processes new or late records without scanning old raw files.
- **Cons:** Risk of double-counting if a batch job fails midway and is re-run without transactional guarantees (requires strict job idempotency).

---

## Strategy 3: Event Time vs. Processing Time Partitioning (Lambda/Kappa Light)

Separate the concept of **when the event occurred** (Event Time) from **when the pipeline ingested it** (Processing Time).

### How It Works:

- **Storage:** Store incoming raw events in HDFS/S3 partitioned by **ingestion date** (`/staging/ingest_date=2026-07-20/`).
- **Processing:** The Spark job reads _only_ the new `ingest_date` folder, extracts the internal `event_date` from each payload, groups the late events by their original `event_date`, and updates those specific historical rows in the MySQL database.

---

## Summary Recommendation for Advertising Analytics

For our specific use case (handling ~1.75M daily events across email and e-commerce):

> 1. **Adopt a 7-Day Sliding Window (Strategy 1):** Since 1.75 million daily records is relatively small for Apache Spark, re-calculating a 7-day window takes negligible compute time while keeping code clean, predictable, and fully idempotent.
> 2. **Partition Source Cache by Event Date:** Ensure custom extractors route records into HDFS folders based on the payload's `event_timestamp`.
> 3. **Set a Hard Cutoff:** Any data arriving **later than 7 days** is logged to an audit table for monitoring and discarded from standard CTR reports, as advertising performance metrics lose business relevance after a week.
