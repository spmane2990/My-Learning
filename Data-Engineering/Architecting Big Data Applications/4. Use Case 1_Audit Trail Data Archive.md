# Use Case 1: Audit Trail Data Archive

## 1. Problem Definition & Goals

### The Problem

ABC Company operates a high-volume e-commerce platform that logs approximately **10 GB of audit trail data per day**.

- **Current State:** Data is hosted in an Oracle RDBMS database.
- **Bottleneck:** Due to high RDBMS storage costs and horizontal scalability limits, logs are aggressively pruned after only **15 days**.
- **Business Need:** Business analysts require access to **3 years of historical data** (approximately **10 TB total**) to analyze long-term customer behavior trends.

### Architecture Goals

- **Cost Efficiency:** Must be highly cost-effective to construct and maintain, as the archive is intended purely for offline, non-critical analytics.
- **Massive Scalability:** System must naturally scale out to ingest, store, and process multiple terabytes of data.
- **Ad-Hoc Querying:** Analysts require standard SQL or SQL-like query interfaces to pull, filter, and summarize historical logs.

---

## 2. Requirements & Baseline Workflow

### Component Analysis

- **Inputs:** A single audit trail table hosted in an Oracle RDBMS. Access is restricted to basic SQL query pulling inside the corporate trust boundary.
- **Input Constraint:** The Oracle database is shared with critical live production systems. The archive pipeline must limit its concurrent connection footprint to avoid performance degradation on active transactional databases.
- **Outputs:** A long-term, low-cost persistent storage tier that surfaces a SQL-friendly query layer. Latency requirements for these analytics queries are highly flexible (no real-time speed constraints).
- **Functional Transformations:** The architecture must periodically read new records, scrub out sensitive user attributes (specifically, **Customer Names**), and push the clean logs to the archive.

### Baseline Functional Workflow

To protect the shared source database, the batch job is set to run periodically (e.g., every hour) to keep individual extraction sizes manageable.

```
[Oracle DB Source]
        │
        ▼
[Fetch Pending Records] ──► (Uses bounded timestamps & tracks last retrieved bookmark)
        │
        ▼
[Data Scrubbing]        ──► (Performs record-by-record processing to strip Customer Names)
        │
        ▼
[Data Archive Target]   ──► (Executes batch inserts into the long-term analytical store)

```

## 3. Designing for Scale

Because every log record is completely self-contained and independent of its neighbors, the data requires no cross-referencing or global complex aggregations. This allows engineers to design for **maximum parallel execution**:

- **Mini-Batch Partitioning:** A single large hourly extraction batch can be broken into small, parallel sub-tasks using distinct high-cardinality attributes (e.g., executing concurrent extractions divided by specific user regions, states, or a hash calculation on the record IDs).
- **Concurrency Support:** The chosen ETL execution framework and target database must natively support high-concurrency writing across multiple distributed nodes out of the box.
- **Source Protection Gate:** The ETL cluster must explicitly limit its maximum concurrent connection pool size against the source Oracle instance to satisfy the live application constraint.

## 5. Final Architecture Review

The finalized analytical batch archiving architecture maps out as a clean, highly parallel distributed system:

### Key Architecture Features

- **End-to-End Concurrency:** Apache Spark uses native database drivers to spin up independent parallel task threads. Each individual task reads a distinct slice of Oracle data (using an ID hashing algorithm) and writes its clean outputs directly to a dedicated partition in HDFS.
- **Date-Based Folder Partitioning:** Data in HDFS is physically organized into directories structured by date (`/archive/year=YYYY/month=MM/day=DD/`). This layout provides natural indexing for analysts' date-range queries, significantly accelerating Apache Impala's scan performance.
- **Simplified Data Pruning:** Physical folder grouping makes historical lifecycle management trivial; purging data beyond the 3-year threshold requires a simple directory drop rather than costly database row deletions.
- **Linear Horizontal Scale:** Both Apache Spark and HDFS scale linearly. As e-commerce volume grows beyond 10 GB/day, capacity can be expanded simply by adding low-cost commodity nodes to the server clusters without altering application code.
