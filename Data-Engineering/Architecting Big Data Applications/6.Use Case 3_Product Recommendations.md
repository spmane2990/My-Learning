# Use Case 3: Product Recommendations

## 1. Problem Definition & Goals

### The Problem

XYZ is a global e-commerce enterprise selling household products to **10 million active users**, processing roughly **200,000 transactions/day**.

To drive revenue and boost engagement, XYZ wants to surface personalized, real-time product recommendations during customer browsing sessions. A Machine Learning (ML) model—developed by the data science team and hosted as an external API—generates these custom recommendation lists using customer demographics and historical transaction behavior.

### Architecture Goals

- **Massive User Scale:** Generate and update recommendations for a base of 10 million active users (and scale beyond as the company grows).
- **Daily Batch Recalculation:** Re-evaluate and refresh recommendations every 24 hours for all users who conducted transactions on the previous day.
- **Rich Context Inputs:** Combine recent transaction history with demographic data to feed the ML model API.
- **Low-Latency Online Serving:** Store pre-computed ranked lists in a high-speed database that supports sub-second, real-time lookups by `Customer ID` during active web sessions.

---

## 2. Requirements & Pipeline Workflow

### Component Analysis

- **Inputs:**
- _Customer Demographics:_ Age, gender, income, etc., stored in a relational database (RDBMS) accessible via SQL.
- _Customer Transactions:_ Live purchase streams (product details, prices, timestamps) published to an **Apache Kafka** topic.

- **Outputs:** A serving database storing ranked product recommendation lists keyed by `Customer ID`, accessible in real time by the e-commerce web application.
- **Non-Functional Requirements:**
- _Retention:_ Store **3 years** of historical transaction logs.
- _Latency:_ Daily recommendation batch updates must finish within **24 hours**.
- _Availability:_ Serving database requires **24/7 high availability and redundancy**; the offline batch pipeline can accept up to 1 hour of maintenance downtime.

### Baseline Functional Workflow

```
[Kafka Topic (Transactions)]  ──► [Transaction Archiver]   ──► [Transaction History DB] ──┐
                                                                                           ├──► [Recommendation Generator] ──► [ML Model API] ──► [Serving DB]
[RDBMS (Demographics)]        ─────────────────────────────────────────────────────────────┘

```

1. **Archive Transactions:** Consume purchase events from the Kafka stream and append them into the long-term **Transaction History DB**.
2. **Identify Active Users:** Identify all customers who completed transactions on the given day.
3. **Enrich & Merge Context:** For each active customer, retrieve historical purchase records from the History DB and combine them with static demographics from the RDBMS.
4. **Inference Call:** Pass the enriched customer payload to the external **ML Model API** to obtain a ranked recommendation list.
5. **Upsert Recommendations:** Save/overwrite the fresh recommendation rankings into the high-speed **Recommendations Serving DB**.

---

## 3. Designing for Scale & Bottlenecks

### Concurrency & Partitioning Strategies

- **Kafka Partitioning by Customer ID:** Using `Customer ID` as the Kafka message key ensures all transactions for a single customer land on the same Kafka partition, preserving event order and allowing parallel task workers to read independent partitions concurrently.
- **Parallel Batch Execution:** The recommendation generator distributes user processing across independent cluster tasks. Each task processes a slice of active users in parallel.
- **Caching Demographics:** Joining transaction logs with an RDBMS on every batch run creates heavy load. Because demographic data changes infrequently, worker nodes can cache demographic tables locally (e.g., using distributed in-memory lookups) to eliminate database join bottlenecks.

### System Scaling Limitations

- **ML API Rate Limits:** The external ML inference API can easily become a major bottleneck if flooded with concurrent requests. The batch pipeline's thread count must be throttled to match the API's maximum supported throughput (as sized by the Data Science team).
- **High-Volume Upserts:** The serving database must support efficient bulk writes/updates without row locking or degradation, as millions of records are refreshed daily.

## 5. Final Architecture Review

The complete product recommendation system orchestrates streaming ingestion, document history archiving, distributed batch processing, and high-speed NoSQL serving:

### Key Architecture Features

- **Streaming Ingestion to Document Archive:** Apache Spark consumes purchase events from Apache Kafka and writes them directly into **MongoDB**. MongoDB's flexible document schema easily nests transaction histories under individual customer records.
- **Decoupled ML API Integration:** Spark handles customer data assembly and executes concurrent HTTP calls to the ML Model API. _(Note: If API latency becomes a bottleneck, the ML scoring code can be embedded directly inside the Spark executor workers)._
- **In-Memory Lookups:** Demographic tables from the RDBMS are cached in memory across Spark executors to speed up user enrichment.
- **Ultra-Fast Serving via Cassandra:** Pre-computed top-N recommendations are bulk-written to **Apache Cassandra**. When a customer browses the website, the web app queries Cassandra directly by `Customer ID` for sub-20ms response times.
- **Lifecycle Management:** Old transactions in MongoDB beyond the 3-year retention threshold are purged via automated TTL indexes or periodic background cleanup jobs.

When deploying Machine Learning models for batch scoring on large datasets using Apache Spark, architects face a fundamental choice: **call an external model via a REST API** or **embed the model directly inside Spark executors**.

This decision impacts processing speed, infrastructure cost, operational independence, and system reliability.

## Architectural Overview

```
Approach 1: External REST API
[Spark Executor Node] ──(HTTP POST)──► [API Gateway / Model Service] ──► [Inference Engine]
                      ◄──(JSON Response)─

Approach 2: Embedded Model
[Spark Executor Node] ──(In-Memory Local Call)──► [Embedded Model Object (ONNX/PMML/MLeap)]

```

## Direct Comparison Matrix

| Architectural Dimension    | External REST API Call                                                           | Embedded Model in Spark Executor                                               |
| -------------------------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Network & I/O Overhead** | **High:** Serialization, HTTP round-trips, and network latency per batch/record. | **Zero Network Overhead:** In-memory execution directly on executor RAM.       |
| **Throughput & Latency**   | Slower for massive batch scoring (e.g., millions of users).                      | **Ultra-Fast:** Maximum throughput; processes millions of records in parallel. |
| **Team Decoupling**        | **High:** Data engineering and Data Science teams operate independently.         | **Low:** Changes to model dependencies require updating Spark build artifacts. |
| **Scalability & Cost**     | Requires scaling both the Spark cluster AND the API hosting infrastructure.      | Scales naturally with the Spark cluster without extra API infrastructure.      |
| **Resource Isolation**     | High: Model memory/GPU demands don't impact Spark JVM memory.                    | Low: Model memory consumption directly competes with Spark execution memory.   |

## 1. External REST API Service

### Pros

- **Clear Team Decoupling:** Data Science teams can update, retrain, or swap the model backend (e.g., in Python/FastAPI or Triton Inference Server) without requiring data engineers to re-deploy or re-compile the Spark pipeline.
- **Reusability Across Channels:** The exact same REST API can serve both **real-time single-user requests** (e.g., live web sessions) and **batch scoring jobs**, ensuring 100% logic consistency between online and offline modes.
- **Resource Isolation:** Heavy model dependencies (such as C++ GPU libraries or TensorFlow runtimes) run outside the Spark JVM, preventing native memory leaks or Out-Of-Memory (OOM) crashes on Spark executors.

### Cons

- **Severe Bottlenecks at Scale:** Sending millions of HTTP requests to a REST API creates network congestion, serialization overhead (JSON parsing), and high latency.
- **Risk of Overwhelming the API:** If Spark spins up hundreds of concurrent worker tasks, it can unintentionally launch a Denial-of-Service (DoS) attack on the model API server unless rate-limiting logic is built into Spark.
- **Higher Infrastructure Costs:** You must maintain and pay for a separate auto-scaling cluster (e.g., AWS SageMaker, Kubernetes) to host the model API.

---

## 2. Embedded Model Inside Spark Executors

### Pros

- **Maximum Performance & Throughput:** Eliminates HTTP overhead and network hops completely. Scores are computed locally in memory at native hardware speeds, making it the ideal choice for massive daily batch processing.
- **Cost Efficiency:** No separate API infrastructure to provision, manage, or scale. Compute costs are tied strictly to the Spark job runtime.
- **Resiliency & Simplicity:** Eliminates an external dependency. If an external API goes down, the batch pipeline fails; an embedded model runs self-contained inside Spark.

### Cons

- **JVM & Dependency Management:** Loading Python/C++ machine learning models (like PyTorch, XGBoost, or Scikit-Learn) into Spark's Java/Scala JVM environment requires native wrappers (e.g., ONNX Runtime, PySpark UDFs, or MLeap).
- **Memory Contention:** Large deep-learning models loaded into executor memory reduce the JVM heap space available for Spark joins, aggregations, and caching, increasing the risk of executor `OutOfMemoryError` crashes.
- **Coupled Release Cycles:** Any update to the machine learning model requires re-packaging or re-deploying the Spark application or updating executor-accessible storage paths.

---

## Summary Recommendation

> **Use an External REST API if:**
>
> 1. The batch dataset is relatively small (e.g., thousands to a few hundred thousand records).
> 2. The model relies on complex native dependencies or GPUs that are hard to run inside Spark JVMs.
> 3. The Data Science and Data Engineering teams need total independence in their release cycles.

> **Use an Embedded Model if:**
>
> 1. You are scoring **millions to billions** of records daily where network latency would cause massive batch run delays.
> 2. The model can be exported to a lightweight runtime format (like **ONNX**, **PMML**, or **Spark MLlib native objects**).
> 3. You want to minimize cloud costs by running scoring within the existing Spark cluster compute allocation.

Handling **cold-start users** (newly registered users or anonymous guests who have zero purchase or interaction history in MongoDB) is a classic problem in recommendation architecture.

Because collaborative filtering algorithms (like Spark MLlib's ALS) require historical interaction vectors to generate personalized scores, a cold-start user will yield an empty result from the ML model.

Here is how the **Product Recommendation Architecture** should handle cold-start users across both the **offline batch processing layer** and the **real-time serving layer**.

---

## The Fallback Hierarchy (Multi-Tier Strategy)

Instead of returning an empty UI component or throwing a 404 error, the serving API should implement a graceful fallback cascade:

```
                  [Web Application / User Request]
                                 │
                                 ▼
                 Does user have profile in Cassandra?
                      ┌──────────┴──────────┐
                     YES                   NO (Cold-Start)
                      │                     │
                      ▼                     ▼
             [Personalized Recs]     Does user have Demographic info?
             (Spark ML Model)              ┌────────┴────────┐
                                          YES               NO
                                           │                │
                                           ▼                ▼
                                 [Demographic Recs]   [Global Popularity]
                                 (Age/Gender Top 10)  (Overall Best Sellers)

```

---

## 1. Demographic & Rule-Based Fallbacks (Mid-Tier)

Even if a user has **no transaction history** in MongoDB, they may have supplied basic information during sign-up (e.g., age, gender, location) that exists in the relational Demographics database.

- **How it works:** During the daily Spark batch run, calculate aggregate "Best Sellers" broken down by demographic segments:

$$\text{Segment Key} = \text{Age Group} + \text{Gender} + \text{Region}$$

- **Storage:** Store these pre-aggregated segment rules in Cassandra under keys like `segment:25-34:F:IN`.
- **Serving Logic:** If the serving API sees a `user_id` missing from the main personalized table, it checks their demographic profile and fetches the top items for that segment.

---

## 2. Global Popularity / Best-Seller Fallback (Baseline Tier)

For completely anonymous users or new sign-ups with zero demographic data, fall back to global platform metrics.

- **How it works:** The Spark batch job runs a simple global aggregation step calculating the **Top 20 Trending/Best-Selling Products** across the entire platform over the past 7 days.
- **Storage:** Store this single array in Cassandra or Redis under a global static key (`global:trending_now`).
- **Serving Logic:** If no personalized or demographic key exists, serve the global trending list instantly. This guarantees **100% SLA uptime** and zero empty recommendation slots on the frontend.

---

## 3. Micro-Batch Real-Time Session Ingestion (Near Real-Time Bridge)

Relying solely on a 24-hour daily batch pipeline means a new user who spends 30 minutes browsing and adding items to their cart won't receive personalized recommendations until the _next day's_ batch run.

To bridge this 24-hour gap:

- **Stream Clicks to Kafka:** As the cold-start user browses, publish real-time clickstream events (`item_viewed`, `added_to_cart`) to the Apache Kafka topic.
- **Item-to-Item Similarity Lookup:** When a cold-start user views Product $A$, bypass the missing _User-Profile_ and query an **Item-Item Similarity Matrix** in Cassandra (`item_id:A -> [similar_items]`). This allows showing _"Customers who viewed this item also viewed..."_ immediately during their very first session.

---

## Architecture Implementation Summary

| User State                     | Secondary Fallback Layer   | Cassandra Key Looked Up     | Latency / Source                         |
| ------------------------------ | -------------------------- | --------------------------- | ---------------------------------------- |
| **Existing Active User**       | Personalized Batch ML      | `user:<user_id>`            | Sub-20ms (Spark ALS Model output)        |
| **New User with Profile**      | Demographic Segment Top 10 | `segment:<age>:<gender>`    | Sub-20ms (Pre-calculated aggregate)      |
| **Anonymous / Unknown**        | Global Best Sellers        | `global:trending_now`       | Sub-5ms (Cached global list)             |
| **Active Session (Cold User)** | Item-Item Similarity       | `item:<current_product_id>` | Sub-20ms (Based on real-time click item) |

> **Key Architecture Takeaway:** The ML model handles personalization for warm users, while the batch pipeline generates **pre-computed aggregate safety nets** (demographic and global trending keys) so the Cassandra serving layer _never_ returns an empty result to the web application.
