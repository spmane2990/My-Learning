# Course Notes: Batch Application Architecture Principles

## 1. Operational Scheduling & Resource Tuning

The choice of schedule interval directly dictates the cost overhead, overall resource utilization, and systemic latency of a batch big data pipeline.

### Balancing Schedule Intervals

- **Small Intervals (e.g., 5 to 30 minutes):** Lowers data latency but introduces steep infrastructure cost penalties. Every batch cycle carries a baseline, fixed computational overhead regardless of the volume of data processed.
- **Large Intervals (e.g., daily):** Minimizes baseline overhead and optimizes hardware efficiency, but results in a significant time delay before business results become available.

### Scheduling Best Practices

- **Stagger Multi-Job Pipelines:** When multiple batch applications share a single hardware cluster (e.g., an Apache Spark cluster), stagger their start times. Keeping the resource pool continuously engaged prevents nodes from competing for compute power simultaneously.
- **Implement Overrun Buffers:** Use historical benchmarks to establish average run times, then inject a safety time buffer before the next batch is slated to begin. This ensures unexpected spikes do not trigger a cascading execution delay.
- **Leave Room for Reprocessing:** Ensure the scheduling window leaves enough idle cluster capacity to completely re-run historical data if an upstream failure occurs, without interrupting current data inflows.

## 2. Volume Minimization & Data Pruning

Because big data scaling demands proportional CPU, memory, network, and disk bandwidth, pruning data as early as possible drastically reduces active infrastructure costs.

Most source feeds are general-purpose datasets (e.g., an orders database containing customer addresses and logistics details). A single analytical pipeline calculating product metrics only requires a fraction of that data.

### Volume Reduction Strategies

```
[General-Purpose Data Source]
         │
         ├──► Strategy A: Source-Level Filter (e.g., SQL WHERE clause / Parquet pushdowns)
         │
         └──► Strategy B: In-Pipeline Filter (Immediate removal of rows/columns)

```

- **Source-Level Pruning:** Request dedicated, pipeline-specific views from data producers. If accessing databases directly, leverage a `WHERE` clause. For HDFS Parquet files, use column projection pushdowns to eliminate unwanted fields before they reach the compute cluster.
- **Early Pipeline Filtering:** If source data cannot be customized, execute row and column filters at step one of the pipeline to protect downstream stages from processing junk data.
- **Isolate Binary Large Objects (BLOBs):** Avoid passing heavy tester images or documents through data streams. Store BLOBs in an underlying file system (like Amazon S3 or HDFS) and pass lightweight object URLs between your compute tasks. Only retrieve the raw asset when it is explicitly needed for processing.

## 3. Distributed Load Balancing & Key Selection

For clusters to scale horizontally when processing heavy grouping or aggregation operations, workloads must be split evenly. Poor distribution strategies cause severe data skew, choking isolated nodes while leaving others completely idle.

### Processing Allocation Archetypes

- **Single-Record Operations:** Functions like `map`, `filter`, or `transform` are completely context-free. They can be safely assigned to nodes on a simple round-robin basis.
- **Key-Based Operations:** Tasks that depend on transaction context require all related events to be directed to the exact same node using an explicit distribution key.

### Choosing the Ideal Distribution Key

The properties of your selection key establish the absolute ceiling of your cluster's ability to scale:

- **High Cardinality:** Ensure the key has thousands of unique values. For instance, using `Customer Type` as a key when only 5 customer categories exist means your pipeline can only ever scale across 5 active nodes. Using `Transaction ID` enables massive, balanced scaling.
- **Even Distribution:** Avoid fields where a single value dominates. If you key data by `Country`, but 50% or more of your transactions originate from a single nation, the node assigned to that country will choke.

## 4. Advanced Performance: Caching & Reprocessing

### Caching for Low-Latency Lookups

Distributed caches (e.g., Redis) sit adjacent to or inside compute pods to deliver ultra-fast, in-memory read access that bypasses heavy disk-based databases.

- **Static & Near-Static Enrichment:** Ideal for hosting lookup records, such as mapping a short coupon code to its corresponding discount percentage, or storing administrative threshold parameters for alerting pipelines.
- **Transient State Tracking:** Excellent for managing global counters or caching intermediate aggregation steps across distinct processing nodes to prevent duplicate compute runs.
- **Constraints:** Architectures must actively account for strict memory size limitations and implement clear cache eviction routines to avoid out-of-date states.

### Designing Idempotent Reprocessing Pipelines

Pipelines inevitably require historical reprocessing due to infrastructure timeouts, application code bugs, late-arriving data packets, or new analytics requests.

1. **Preserve Raw Source Ingestion:** 7+ Day Hold.
   Never prune or delete raw source inputs immediately after a batch completes. If historical late data can arrive up to a week later, retain the original inputs for at least a 7-day safety window.

2. **Enforce Deterministic Re-runs:** Time-Bound Boundaries.
   Design batches to be repeatable and bound strictly by clear date-and-time intervals. Executing the same batch interval multiple times must always capture the exact same records and yield identical outputs.

3. **Eliminate Increment Counters:** Avoid State Accumulators.
   Never rely on basic accumulators or increment counters, which cause catastrophic double-counting if a batch is re-run.

4. **Recompute via Batch Isolation:** Delta Updates.
   Structure pipelines to completely clear out and recalculate data at the batch level. This allows engineers to refresh only the specific partition keys that changed rather than reprocessing the entire historical database from scratch.
